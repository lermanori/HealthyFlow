### #9 — Drag-and-drop reorder not persisting (Backlog)
**RCA complete.** Root cause: `position` field does not exist anywhere in the stack.
- Not in DB schema
- Not in `Task` type
- Not in backend route
- Not in frontend mutation
- `GET /tasks` always re-sorts by `start_time → created_at` on every fetch, overwriting any client-side order.

**Architecture decision open:**

| Option | Approach | Trade-off |
|---|---|---|
| Manual display order | New `position INTEGER` column + `PATCH /tasks/reorder` endpoint | Atomic, single round-trip; position is independent of time |
| Reschedule gesture | Drag updates `startTime` | No schema change needed; conflates visual order with scheduling |

**Preferred:** `PATCH /tasks/reorder` (atomic bulk update). Full RCA discussion in Code tab session "RCA: drag-and-drop not persisting".

## Architectural decisions reinforced today

- **Zod as single source of truth** — all data shapes defined as Zod schemas; `z.infer<>` for TypeScript types. Already in place; reinforced in CLAUDE.md.
- **Server-keyed OpenAI only** — no BYOK, no client-side key handling. No silent fallbacks; errors surface to the caller via `Result<T>`.
- **Deep modules** — business logic lives in fat service files (`openai.ts`, `rollover.ts`), not thin per-feature files.
- **Virtual-first habit instances** — instances synthesized at query time; only materialized to DB on completion or drag. See `docs/adr/0001-materialize-habit-instance-on-drag.md`.
- **Thin routes** — route handlers validate (Zod) + call service + return. No logic in route files.
